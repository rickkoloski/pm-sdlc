# Prototyping Task Descriptions

Descriptions for each moment, keyed by `moment.{key}`. Unlike other lifecycles, these are not temporal divisions — they're states you recognize as they happen. Move each task to "In Progress" when you notice you're in that moment.

### moment.spark
Start building. An idea takes shape and the fastest way to explore it is to write code. No spec, no plan, no ceremony needed. The disciplines are a toolkit you can reach for if friction emerges, but right now the goal is momentum. This is the on-ramp — every prototype starts here.

### moment.friction
Something isn't working or isn't clear. You keep changing your mind, you're building the same thing three ways, or you can't explain what you're building. This is the signal to reach for a discipline: research if you don't know what to build, analysis if requirements are fuzzy, architecture if the structure is fighting you, testing if you're not sure it works. Friction isn't failure — it's the prototype telling you where to add structure.

### moment.traction
The work is clearly going somewhere. You can articulate what is being built and why. The core concept is demonstrable. This is the moment to consider lightweight structure: a retroactive spec, a few architectural decisions written down, tests on the critical path. Not because process demands it, but because the work has earned it.

### moment.commitment
The prototype is becoming the real thing. This is the transition from exploratory to intentional. Write a retroactive spec if one doesn't exist. Add test coverage on core paths. Identify a deployment strategy. Choose the target lifecycle for ongoing work (Scrum, RUP, Kanban). The disciplines that were optional are now essential — architecture, testing, design, analysis, and deployment all activate here.

### moment.reentry
Bringing breakout work back into a structured lifecycle. Document findings and communicate them to the parent lifecycle. Map what was built to the target lifecycle's expectations (specs, stories, architectural decisions). Decide what to keep vs. discard. This moment completes the prototype's journey — the learning flows back into organized work, and the temporary freedom of prototyping mode ends.
